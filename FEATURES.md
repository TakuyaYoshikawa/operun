## 1. 認証・マルチテナント

### 概要
複数の会社が同一サービスを利用するSaaS構成。会社ごとにデータが完全分離される。

### 機能詳細
- **テナント登録**（`POST /api/auth/register`）
  - 会社名・メールアドレス・パスワードで新規テナントを作成
  - 初回プランは `trial`（trial / light / standard / pro）
- **ログイン**（`POST /api/auth/login`）
  - メールアドレス＋パスワードで JWT トークンを発行
  - トークンに `tenant_id` を含める
- **データ分離**
  - 全テーブル（Machine / Process / Order / Operation）に `tenant_id` を持つ
  - 全 API で `tenant_id` フィルタ必須。他テナントのデータは参照・変更不可

### デモアカウント
| 項目 | 値 |
|---|---|
| Email | demo@example.com |
| Password | demo1234 |
| 会社名 | 山田製作所 |

## 2. 設備・工程マスタ管理

### 設備マスタ（`/api/machines`）
| フィールド | 説明 |
|---|---|
| 設備コード | テナント内で一意（例：M01） |
| 設備名 | 表示名（例：旋盤1号機） |
| 1日稼働時間 | デフォルト 8.0 時間 |
| 段取り時間 | 標準段取り時間（分） |

### 工程マスタ（`/api/machines/processes/`）
| フィールド | 説明 |
|---|---|
| 工程コード | テナント内で一意（例：P01） |
| 工程名 | 表示名（例：旋削、フライス加工） |
| 標準時間 | 単位あたり標準加工時間（分/個） |

- CRUD 操作（一覧・登録・更新・削除）

## 3. 受注管理

### 受注（`/api/orders`）
| フィールド | 説明 |
|---|---|
| 受注番号 | テナント内で一意（例：ORD-001） |
| 品名 | 製品名 |
| 品番 | 製品コード |
| 数量 | 製造数量 |
| 納期 | 完成期限日 |
| 優先度 | 1=特急 / 2=高 / 3=通常 |
| ステータス | pending / in_progress / done |
| 備考 | 自由記述 |

- CRUD 操作（一覧・登録・更新・削除）
- ステータス・優先度でフィルタリング可能
- 削除時は紐づく工程も cascade 削除

### スケジューリング優先順
1. 特急フラグ（is_urgent = True を最優先）
2. 優先度（数字が小さいほど高い）
3. 納期（早い順）

## 4. 複数工程スケジューリング

### 概要
1つの受注に複数の工程（Operation）を設定できる。各工程を異なる設備に割り当て、前工程が終わってから次工程が始まる依存関係を自動で考慮する。

### 工程（`/api/orders/{order_id}/operations`）
| フィールド | 説明 |
|---|---|
| sequence | 工程順序（自動採番） |
| 設備 | 担当設備（machineから選択） |
| 工程種別 | 工程マスタから選択（任意） |
| 所要時間 | 作業時間（時間単位） |
| 特急フラグ | この工程を最優先扱いにする |

- **sequence 自動採番**：既存最大値 + 1 で連番付与
- **削除時の詰め直し**：途中の工程を削除すると後続の sequence が自動的に繰り上がる
- **依存関係**：工程2は工程1の計画終了後にしか開始しない

### 例：3工程の受注

```
受注 ORD-001「シャフト部品A」
  工程1: 旋盤1号機 → 旋削 → 4.0h
  工程2: マシニング1号機 → マシニング → 2.0h
  工程3: 研削盤1号機 → 研削 → 1.5h
```

ガントチャート上でそれぞれの設備行にバーが並ぶ。

## 5. ガントチャート

### 概要
全受注の工程を設備ごとに横軸（日単位）に自動配置して可視化する。

### 表示仕様
| 色 | 意味 |
|---|---|
| 緑 | 通常受注（納期内） |
| オレンジ | 特急受注 |
| 赤 | 納期超過（完了予定が納期を過ぎている） |

- **再スケジュールボタン**：受注や工程を変更したあとに押すと計画日時を再計算
- **土曜列**：薄い青背景で表示
- **日曜列**：薄い赤背景（非稼働扱い）
- **遅延サマリ**：画面下部に納期超過受注の一覧を表示

### スケジューリングアルゴリズム
- EDD（Earliest Due Date）フォワードスケジューリング
- 設備の稼働カレンダーを考慮（日跨ぎ・日曜除外）
- Phase 3 で OR-Tools による数理最適化エンジンに差し替え予定

## 6. 納期シミュレーター（F-05）

### 概要
新規受注が入ったとき、現在の計画に差し込んだ場合の**完成予定日**と**既存受注への影響**を即座に回答する。営業担当が顧客からの電話問い合わせに30秒で答えられる。

### 入力項目
| 項目 | 説明 |
|---|---|
| 品名 | 任意（試算用） |
| 使用設備 | どの設備で加工するか |
| 所要時間 | 加工にかかる時間（時間） |
| 希望納期 | 顧客から要求された納期 |
| 優先度 | 通常 / 高 / 特急 |

### レスポンス内容
| 項目 | 説明 |
|---|---|
| 受注可否 | 納期内に完成するか |
| 完成予定日 | 「2026年4月8日（水）」形式 |
| 所要営業日数 | 今日から完成までの営業日数 |
| 影響受注 | 後ろ倒しになる既存受注の一覧と件数 |

### エンドポイント
```
POST /api/schedule/simulate/delivery
```

## 7. CSVインポート

既存 Excel データを CSV に変換してそのまま一括登録できる。upsert 方式のため、再実行しても重複しない。

### 受注インポート（`POST /api/csv/orders`）
```csv
order_number,product_name,product_code,quantity,due_date,priority,note
ORD-001,部品A,ABC-001,50,2026-04-30,3,
ORD-002,部品B,ABC-002,20,2026-05-15,2,急ぎで
```

### 設備マスタインポート（`POST /api/csv/machines`）
```csv
code,name,daily_capacity_hours,setup_time_minutes
M01,旋盤1号機,8.0,30.0
M02,マシニング1号機,8.0,45.0
```

### 工程マスタインポート（`POST /api/csv/processes`）
```csv
code,name,standard_time_per_unit
P01,旋削,12.0
P02,フライス加工,18.0
```

- 文字コード：UTF-8（BOM付き対応）
- 既存レコードはキー（コード・受注番号）で照合して上書き

## 10. 稼働カレンダー

### 概要
日曜だけでなく、日本の祝日・お盆・年末年始・工場独自の休日をスケジューリングに反映する。現行は日曜除外のみのため、GW明けに計画がズレる問題を解消する。

### 休日の種類
| 種別 | 説明 | 取得方法 |
|------|------|---------|
| 日本の祝日 | 国民の祝日 | `jpholiday` ライブラリで自動取得 |
| 工場独自休日 | お盆・年末年始・創立記念日など | 管理画面から手動登録 |
| 土曜設定 | 全休 / 半日稼働（4h）/ 通常稼働 | テナント設定で選択 |

### 工場カレンダー（`/api/calendar`）
| フィールド | 説明 |
|---|---|
| date | 対象日（YYYY-MM-DD） |
| is_holiday | 休日フラグ |
| holiday_name | 休日名（例：お盆休み） |
| working_hours | 稼働時間（0=全休 / 4=半日 / 8=通常） |

- テナントごとに独自カレンダーを保持
- 年単位での一括登録（翌年分を年初に登録するフロー）
- 祝日は `jpholiday` で自動生成し、管理画面で確認・修正可能

### APIエンドポイント追加
| メソッド | パス | 説明 |
|---|---|---|
| GET | /api/calendar | カレンダー一覧（期間指定） |
| POST | /api/calendar/holidays | 休日登録 |
| DELETE | /api/calendar/holidays/{date} | 休日削除 |
| POST | /api/calendar/generate | 祝日自動生成（年指定） |
| GET | /api/settings/calendar | テナントカレンダー設定取得 |
| PUT | /api/settings/calendar | テナントカレンダー設定更新（土曜設定など） |

## 11. 実績入力・進捗管理

### 概要
計画（スケジューリング結果）に対して、実際の作業開始・完了・所要時間を記録する。「今どこまで進んでいるか」をリアルタイムに把握し、計画と実績のズレを可視化する。

### 実績ログ（`/api/operations/{op_id}/logs`）
| フィールド | 説明 |
|---|---|
| operation_id | 対象工程 |
| status | not_started / in_progress / done / on_hold |
| started_at | 実際の開始日時 |
| finished_at | 実際の完了日時 |
| actual_hours | 実際にかかった時間（時間単位） |
| worker | 作業者名（任意） |
| note | 作業メモ（不具合・段取り変更など） |

### ガントチャートへの追加表示
| 表示要素 | 説明 |
|---------|------|
| 計画バー | 既存のバー（薄い色） |
| 実績バー | 計画バーに重ねて表示（濃い色） |
| 進捗率 | バー内に「75%」などを表示 |
| 遅延アラート | 実績が計画を超過した場合に赤枠 |

### 計画 vs 実績サマリ
- 受注ごとの計画工数 vs 実績工数の差異
- 設備ごとの稼働率（計画稼働時間 / 実稼働時間）
- 完了済み受注の平均遅延日数

### APIエンドポイント追加
| メソッド | パス | 説明 |
|---|---|---|
| GET | /api/orders/{id}/operations/{op_id}/logs | 実績一覧 |
| POST | /api/orders/{id}/operations/{op_id}/logs | 実績登録 |
| PUT | /api/orders/{id}/operations/{op_id}/logs/{log_id} | 実績更新 |
| POST | /api/orders/{id}/operations/{op_id}/start | 作業開始（started_at を現在時刻でセット） |
| POST | /api/orders/{id}/operations/{op_id}/finish | 作業完了（finished_at・actual_hours を自動計算） |
| GET | /api/reports/performance | 計画 vs 実績サマリ |

## 12. AI自然言語入力

### 概要
チャット形式でシステムを操作できるAIアシスタント。ITリテラシーが低い現場担当者でも「話しかけるだけ」で受注登録・納期確認・状況把握ができる。使用モデル：Claude Haiku（低コスト・高速）。

### 機能一覧

#### 12-1. 自然言語→受注登録

**フロー**
```
ユーザー入力（自然言語）
  ↓
Claude Haiku で解釈・構造化
  ↓
不明フィールドがあれば聞き返す（会話継続）
  ↓
全フィールド揃ったら受注フォームに自動セット
  ↓
ユーザーが確認・修正のうえ [保存] ボタンを押す
```

**必須フィールド（揃うまで聞き返す）**
- `product_name`：品名
- `quantity`：数量
- `due_date`：納期（「来月末」「明日」などの相対表現も解釈）

**任意フィールド（なければそのままセット）**
- `product_code`：品番
- `priority`：優先度（「急ぎ」→2、「特急」→1、未指定→3）
- `customer_id`：顧客（顧客名で検索・照合）
- `note`：備考

**入力例と解釈例**
```
入力: 「鈴木商事からシャフトA 50個、4/30まで、急ぎで」
解釈:
  product_name: シャフトA
  quantity:     50
  due_date:     2026-04-30
  priority:     2（高）
  customer:     鈴木商事（顧客マスタを検索）
```

**聞き返し例**
```
入力: 「シャフトA 急ぎで」
応答: 「数量と納期を教えてください。」

入力: 「30個、来月末」
応答: 「以下の内容をフォームにセットします。確認・修正のうえ保存してください。」
→ フォームに自動セット
```

#### 12-2. 納期回答の自然言語化

F-05（納期シミュレーター）のJSONレスポンスをClaude Haikuが日本語に変換する。営業担当が電話口でそのまま読み上げられる形式。

**変換例**
```
F-05レスポンス（JSON）→
「5月2日（木）に完成できます。
 既存のORD-003とORD-007が1日ずつ後ろ倒しになります。
 お客様への納期は5月2日でお伝えください。」
```

#### 12-3. 状況確認チャット

ガントデータをコンテキストに渡して自然言語でQA。

**想定クエリ例**
```
「今週遅延しそうな案件は？」
「旋盤1号機の来週の空き時間は？」
「特急案件は今いくつある？」
「鈴木商事の受注はどうなってる？」
```

**実装上の注意**
- ガントデータは全件ではなく「今週分」「特急のみ」などフィルタしてから渡す（トークン節約）

## 13. 品番テンプレート（簡易BOM）

### 概要
品番ごとに「標準工程セット」を登録しておき、受注登録時に品番を選ぶと工程が自動セットされる機能。繰り返し受注品の登録工数を大幅削減する。フルBOMは複雑すぎるため、工程テンプレートに絞った簡易版とする。

### 品番テンプレート（`/api/product-templates`）
| フィールド | 説明 |
|---|---|
| product_code | 品番（例：ABC-001） |
| product_name | 品名（例：シャフトA） |
| standard_operations | 標準工程リスト（JSON配列） |

### 標準工程リストの構造
```json
[
  {
    "sequence": 1,
    "machine_code": "M01",
    "process_code": "P01",
    "hours_per_unit": 0.08
  },
  {
    "sequence": 2,
    "machine_code": "M03",
    "process_code": "P03",
    "hours_per_unit": 0.04
  }
]
```

`hours_per_unit`（単位あたり時間）× 受注数量 = 所要時間として自動計算。

### 受注登録時のフロー
```
品番「ABC-001」を入力
  ↓
テンプレートが存在する場合:
  「前回と同じ工程（旋削→マシニング→研削）を使いますか？」と提案
  ↓
[はい] を押すと数量に応じた所要時間で工程を自動セット
  ↓
[カスタマイズ] を押すと工程を個別に編集可能
```

### AI入力との連携
```
入力: 「シャフトA ABC-001 30個 5月末」
  ↓
品番テンプレートを自動参照
  ↓
「ABC-001の標準工程（旋削4h・マシニング2h・研削1.5h）を
 30個分でセットします。よろしいですか？」
```

## 16. スケジューリング最適化エンジン

### 概要
現行のEDD（Earliest Due Date）フォワードスケジューリングをOR-Toolsによる数理最適化エンジンに差し替える。設備稼働率の最大化・納期遵守率の向上・特急案件の自動優先配置を実現する。

### 現行との比較
| 項目 | 現行（EDD） | Phase 3（OR-Tools） |
|------|-----------|-------------------|
| アルゴリズム | 順番に詰めるだけ | 制約充足問題として最適解を探索 |
| 設備稼働率 | 考慮しない | 最大化を目的関数に含める |
| 複数設備の同時最適化 | 逐次処理 | 全設備を同時に最適化 |
| 計算時間 | 即時 | 数秒〜30秒（件数による） |

### 最適化の目的関数
1. 納期超過ペナルティの最小化（最優先）
2. 設備稼働率の最大化
3. 段取り時間の最小化（同一工程を連続配置）

### 制約条件
- 各設備は同時に1工程しか処理できない
- 工程の依存関係（前工程完了後に次工程開始）
- 稼働カレンダーの考慮
- 特急フラグの優先配置

### フォールバック設計
最適化に30秒以上かかる場合は現行EDDの結果を返し、バックグラウンドで最適化を継続する。

### APIエンドポイント変更
```
POST /api/schedule/run
  Request body に optimizer: "edd" | "ortools" を追加
  デフォルト: "ortools"（Phase 3以降）
```

## 17. 原価管理

### 概要
受注ごとに見積原価（材料費＋加工費）と実際原価（実績工数×時間単価）を比較し、赤字受注の検知と価格設定の改善を支援する。

### 設備マスタへの追加フィールド
```
hourly_rate: float   ← 時間単価（円/時間）
```

### 受注の原価計算
| 項目 | 計算方法 |
|------|---------|
| 見積材料費 | 品番テンプレートの材料 × 単価 |
| 見積加工費 | 工程の標準時間 × 設備時間単価 |
| 見積原価 | 見積材料費 + 見積加工費 |
| 実際加工費 | 実績工数 × 設備時間単価 |
| 実際原価 | 実際材料費 + 実際加工費 |
| 差異 | 見積原価 − 実際原価 |

### 原価レポート
- 受注別の見積 vs 実際の比較表
- 赤字受注の一覧（差異がマイナスの受注を赤表示）
- 設備別・月別の稼働コスト集計

### APIエンドポイント追加

## 21. AIエージェント操作（全機能のチャット統合）

### 概要
すべてのデータ入力・操作をAIチャット画面から自然言語で指示できる「AIエージェントモード」。
現在のAIアシスタントはチャットと受注入力のみ対応しているが、本機能で**マスタ登録・在庫操作・発注管理・スケジュール実行**など全操作をカバーする。
ITリテラシーが低い担当者でも、話しかけるだけでシステムを完全に操作できる状態を目指す。

### 実現技術：Claude Tool Use（関数呼び出し）
Claude APIの「ツール使用」機能を使う。APIリクエストに「呼び出せるツール（＝既存のREST APIの定義）」を渡すと、Claudeが自然言語から意図を読み取り、適切なAPIを自動選択・実行する。

```
ユーザー: 「S45C丸棒φ50を100kg入荷した」
  ↓ Claudeが意図を解釈
  ↓ tools: [receive_stock(material_id, quantity)] を選択
  ↓ バックエンドAPI呼び出し
  → 「S45C丸棒φ50 100kgの入荷を登録しました。在庫: 350kg → 450kgです。」
```

### 対応する操作カテゴリ

| カテゴリ | 指示例 | 呼び出しAPI |
|---------|--------|-----------|
| 受注登録 | 「鈴木商事からシャフトA 50個、来月末」 | POST /api/orders |
| 受注変更 | 「ORD-042の納期を5月10日に変更して」 | PUT /api/orders/{id} |
| 在庫入荷 | 「S45C丸棒 100kg入荷した」 | POST /api/materials/{id}/receive |
| 在庫払出 | 「アルミ丸棒 20本使った」 | POST /api/materials/{id}/issue |
| 発注登録 | 「山田商会にSUS304 50kgを来週末納入で発注して」 | POST /api/purchase-orders |
| 入荷確認 | 「PO-2026-003が届いた、全量受入」 | POST /api/purchase-orders/{id}/receive |
| 設備登録 | 「旋盤3号機を追加して、8時間稼働」 | POST /api/machines |
| 顧客登録 | 「田中精機、担当:田中さん、03-1234-5678を登録して」 | POST /api/customers |
| スケジュール実行 | 「スケジュールを再計算して」 | POST /api/schedule/run |
| 状況確認 | 「今週遅れそうな受注は？」 | /api/ai/gantt-summary + chat |

### 操作フロー（確認ステップあり）

```
ユーザー: 「S45C丸棒φ50を100kg入荷した」
  ↓
AI: 「以下の入荷を登録します。よろしいですか？
      材料: S45C丸棒φ50（MAT-001）
      数量: 100 kg
      [確認] [キャンセル]」
  ↓ [確認] を押す
AI: 「登録しました。在庫: 350kg → 450kgになりました。」
```

破壊的操作（削除・大量変更）は必ず確認ステップを挟む。
読み取り操作（在庫確認・スケジュール照会）は即時回答。

### 実装方針

**バックエンド（/api/ai/agent）**
- 既存の `/api/ai/chat` を拡張
- Claude APIのリクエストに `tools` パラメータで全API定義を渡す
- Claudeが `tool_use` を返したら対応するAPIを呼び出し、結果をClaude に返す（マルチターン）
- 最終的なテキスト回答をフロントに返す

**フロントエンド**
- 現在のAIチャット画面に「エージェントモード」トグルを追加
- 確認ダイアログを会話中にインラインで表示
- 操作結果（登録内容・変更内容）をカード形式で表示

### コスト試算（Claude Haiku）

| 操作種別 | 1回あたり | 備考 |
|---------|---------|------|
| 単純な登録操作 | 約0.3〜0.5円 | ツール定義のトークンが増加するため |
| 複数ステップの操作 | 約0.5〜1.0円 | マルチターン分 |
| 状況確認 | 約0.3円 | ガントデータフィルタあり |

月間1,000操作で約300〜500円。Phase 2と同様、APIコストはサービス価格に対して無視できる水準。

### 制約・注意点
- **ツール定義のトークン数**：全API定義を毎回渡すためinputトークンが増加。頻出ツールに絞るか、コンテキストに応じてツールを動的に選択する最適化が必要
- **曖昧な指示への対応**：「材料コードMAT-001」ではなく「S45C丸棒」のような表記ゆれを吸収するため、材料名での検索（あいまい検索）をツール定義に含める
