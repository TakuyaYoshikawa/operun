## 1. 認証・マルチテナント

### 概要
JWT Bearer トークンによる認証。テナント（会社）ごとにデータを完全分離。

### 仕様
- ログイン：メールアドレス＋パスワード → JWT トークン発行
- トークン有効期限切れ時：自動ログアウト＆リロード（401インターセプター）
- 全APIはテナントIDでフィルタリング（他社データへのアクセス不可）
- ユーザーはテナントに紐づく（将来：複数ユーザー対応）

### データモデル
| テーブル | 主なカラム |
|---------|-----------|
| tenants | id, name, plan |
| users | id, tenant_id, email, hashed_password, name |

### API
| メソッド | パス | 説明 |
|---------|------|------|
| POST | /api/auth/login | ログイン（フォームデータ） |
| GET | /api/auth/me | 現在のユーザー情報 |

## 2. 設備マスタ管理

### 概要
加工設備の登録・管理。設備グループによる自動割り付けと外注設備の管理をサポート。

### 仕様
- 設備コード（テナント内一意）、名称、稼働時間、段取り時間を管理
- **設備グループ（machine_type）**：同じグループの設備はスケジューリング時に自動選択（最も早く空く設備を採用）
- **稼働停止フラグ（is_active）**：停止中の設備はスケジューリング対象外
- **外注設備フラグ（is_outsource）**：外注先名・標準リードタイムを設定可能
- **設備グループのオートコンプリート**：既存グループ名を入力中に候補表示
- **削除保護**：工程実績に使用中の設備は削除不可（エラーメッセージ表示）

### データモデル（machines）
| カラム | 型 | 説明 |
|--------|-----|------|
| name | string | 設備名（例：旋盤1号機） |
| code | string | 設備コード（例：M001） |
| daily_capacity_hours | float | 1日の稼働時間（デフォルト：テナント設定値） |
| setup_time_minutes | float | 段取り時間（分）※スケジューリングに加算 |
| machine_type | string | 設備グループ名（例：旋盤） |
| is_active | bool | 稼働中フラグ |
| is_outsource | bool | 外注設備フラグ |
| outsource_supplier | string | 外注先名 |
| outsource_lead_days | int | 標準リードタイム（日） |

### API
| メソッド | パス | 説明 |
|---------|------|------|
| GET | /api/machines | 設備一覧 |
| POST | /api/machines | 設備登録 |
| PUT | /api/machines/{id} | 設備更新 |
| DELETE | /api/machines/{id} | 設備削除（使用中は400エラー） |

## 9. 受注管理

### 概要
製造受注の登録・管理。工程展開とスケジューリングのトリガー。

### 仕様
- 受注番号（テナント内一意）、品名・品番、数量、納期、優先度、顧客を管理
- 優先度：1=特急、2=高、3=通常
- ステータス：pending（未着手）/ in_progress（進行中）/ done（完了）
- 受注に対して複数の工程（Operation）を登録（順序あり）
- 受注削除時は紐づく工程も一括削除（cascade）
- CSVインポート対応（受注・工程のバルク登録）

### データモデル（orders）

| product_name | string | 品名 |

| quantity | int | 数量 |
| due_date | date | 納期 |
| priority | int | 優先度（1/2/3） |
| status | string | ステータス |
| customer_id | int | 顧客ID（nullable） |

### API

## 10. スケジューリングエンジン

### 概要
受注工程を設備に自動割り付けしてスケジュールを生成。

### アルゴリズム
- **EDD（Earliest Due Date）フォワードスケジューリング**
  - ソート順：特急フラグ → 優先度 → 納期 → 受注ID → 工程順序
  - 設備の空き時刻から順に割り付け（日またぎ・休日・非稼働時間考慮）

- **OR-Tools 最適化**（`/schedule/run` のデフォルト）
  - Google OR-Toolsの CP-SAT ソルバーによる数理最適化

### 主要仕様
- **段取り時間の考慮**：`operation.duration_hours + machine.setup_time_minutes / 60` で実効時間を計算
- **設備グループ対応**：同一グループ内の設備から最も早く空くものを自動選択（machine_locked=Falseの場合）
- **工程順序の依存関係**：同一受注の工程は前工程終了後に次工程を開始
- **稼働カレンダー対応**：休日・非稼働時間をスキップ
- **テナント設定の稼働時間**：稼働開始時刻・1日稼働時間を参照

### 下書きスケジュール機能
スケジュールは現行（確定済み）と下書きの2段階で管理。

| 操作 | 説明 |
|------|------|
| 下書き作成 | 現行スケジュールをそのまま下書きカラムにコピー |
| スケジュール最適化 | OR-Toolsで最適化した結果を下書きカラムに書き込み |
| 下書き編集 | 個別工程の日時・設備を手動変更（PATCH /schedule/draft/{id}） |
| 確定 | 下書き → 現行スケジュールに反映、下書きを削除 |
| 破棄 | 下書きを削除（現行スケジュールはそのまま） |

### API
| メソッド | パス | 説明 |
|---------|------|------|
| POST | /api/schedule/run | スケジュール最適化（下書きに保存） |
| POST | /api/schedule/create-draft | 現行スケジュールを下書きにコピー |
| PATCH | /api/schedule/draft/{op_id} | 下書き工程の手動編集 |
| POST | /api/schedule/commit | 下書きを確定 |
| POST | /api/schedule/discard | 下書きを破棄 |
| GET | /api/schedule/gantt | ガントチャート用データ（draft パラメータで切替） |
| POST | /api/schedule/simulate/delivery | 納期シミュレーション |

## 11. ガントチャート

### 概要
スケジュールを設備行×時間軸で可視化。現行・下書きの切替と手動編集をサポート。

### 表示仕様
- **X軸**：日単位（デフォルト）または時間単位（8時〜稼働時間分）を切替可能
- **Y軸**：設備ごとに1行
- **バーの色**：受注ごとに異なる色（12色パレットから循環）
- **バーのラベル**：品名のみ表示
- **特急マーカー**：`!` アイコン表示
- **納期超過**：赤い枠線で表示
- **土日列**：土曜=青背景、日曜=赤背景
- **ホバーツールチップ**：品名・受注番号・設備・開始/終了・納期・優先度を表示

### 下書きモード
- 下書きがある場合、現行/下書きの切替タブを表示
- 下書きモードではバーに破線アウトラインと ✎ アイコン
- バークリックまたはテーブルの編集ボタンで日時・設備を変更するモーダルを表示
- 変更は即座にDBに保存（中断しても保持）

### スケジュール一覧テーブル
ガントチャート下部に工程単位のテーブルを表示。
- 列：受注番号 / 品名 / 設備 / 開始予定 / 終了予定 / 作業時間 / 納期 / 状態
- 開始予定順でソート
- 下書きモード時は各行に「✎ 編集」ボタンを表示

### 凡例
- オーダー凡例チップ（色見本＋受注番号＋品名）を上部に表示（最大12件）
- 納期超過受注のサマリーを下部に表示

## 12. 工程実績入力（現場向け）

### 概要
スマートフォン等での現場からの工程実績入力を想定したUI。

### 仕様
- **対象工程**：当日（または直近の稼働日）の工程を自動表示
- **設備フィルター**：担当設備で絞り込み
- **日付切替**：今日 / 日付指定で切替
- **自動更新**：30秒ごとに最新情報を取得
- **ステータス管理**：not_started → in_progress → done / on_hold
- **経過時間表示**：進行中の工程は開始からの経過時間をリアルタイム表示
- **完了時の自動処理**：全工程完了で受注ステータスを自動的に done に変更

### ステータス遷移
| 操作 | 遷移前 | 遷移後 |
|------|--------|--------|
| 開始 | not_started | in_progress |
| 完了 | in_progress | done（実績時間を自動計算） |
| 保留 | in_progress | on_hold（理由を記録） |
| 完了済の表示切替 | — | showDone トグルで表示/非表示 |

### API
| メソッド | パス | 説明 |
|---------|------|------|
| GET | /api/operations/today | 本日の工程一覧 |
| GET | /api/operations | 工程一覧（フィルタ可） |
| GET | /api/operations/{id} | 工程詳細 |
| POST | /api/operations/{id}/start | 開始 |
| POST | /api/operations/{id}/complete | 完了 |
| POST | /api/operations/{id}/hold | 保留 |
| PATCH | /api/operations/{id}/note | メモ更新 |

## 14. AIアシスタント

### 概要
Claude（Anthropic）を活用したAIアシスタント。3つのモードで利用可能。

### モード

#### チャットモード
- スケジュール状況の質問に日本語で回答
- ガントサマリーをコンテキストとして自動付与
- 使用モデル：claude-haiku-4-5

#### 受注登録モード（自然言語入力）
- 「部品A 50個 来月末まで」のような自然言語テキストから受注フォームを自動生成
- 複数受注の一括解析に対応
- 不足フィールドを missing_fields として返却

#### エージェントモード
- ツールを呼び出してシステムを直接操作
- **確認フロー**：書き込み操作は実行前にユーザーに内容を提示し、承諾後に実行
- 使用モデル：claude-sonnet-4-6

### エージェント利用可能ツール

| ツール | 操作 |
|--------|------|
| search_orders | 受注を検索・確認 |
| **update_order** | 受注の納期・優先度・数量・ステータス・備考を変更 |
| **create_order** | 新規受注を登録 |
| search_materials | 材料・在庫を検索 |
| receive_stock | 材料の入庫登録 |
| issue_stock | 材料の払出登録 |
| create_purchase_order | 材料の発注を登録 |
| search_customers | 顧客を検索 |
| create_customer | 顧客を登録 |
| get_schedule_summary | スケジュール状況サマリーを取得 |
| run_schedule | スケジュールを再実行 |

### 操作例
```
ユーザー：「ORD-007の納期を5月25日に変更してください」
AI：「ORD-007（スリーブ部品G）の納期を4/8 → 5/25に変更します。よろしいですか？」
ユーザー：「はい」
AI：「納期を変更しました。スケジュールを再実行しますか？」
```

### API
| メソッド | パス | 説明 |
|---------|------|------|
| POST | /api/ai/chat | チャット |
| POST | /api/ai/parse-order | 自然言語→受注データ変換 |
| POST | /api/ai/agent | エージェントモード |
| POST | /api/ai/explain-simulation | 納期シミュレーション結果の文章化 |
| GET | /api/ai/gantt-summary | ガントサマリー取得 |

## 17. 安全性・エラーハンドリング

### データ整合性
- **マスター削除保護**：使用中の設備・工程・顧客は削除不可（400エラーを返し、UIにメッセージ表示）
- **受注削除時のカスケード**：受注削除時は紐づく工程も一括削除
- **SQLiteマイグレーション**：Alembic未使用のため `PRAGMA table_info` + `ALTER TABLE ADD COLUMN` でカラム追加

### フロントエンド安全性
- **ErrorBoundary**：コンポーネントのレンダリングエラーをキャッチし、再読み込みボタンを表示
- **401インターセプター**：トークン期限切れを検知し、自動ログアウト＆リロード
- **throwOnError: false**：TanStack Queryのクエリエラーが ErrorBoundary を誤発火させない設定

### 認証
- リクエストインターセプターでlocalStorageからトークンを毎回読み込み（リロード時の認証漏れ防止）
